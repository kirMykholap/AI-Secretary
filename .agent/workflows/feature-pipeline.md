# Feature Pipeline

Этот Workflow организует последовательный процесс решения сложной задачи. 
В нем учитывается, что для каждой фазы ИИ временно принимает (примеряет) новую роль, забывая о других обязанностях, чтобы сфокусироваться.

## Алгоритм пайплайна
При вызове этого workflow агенту нужно строго следовать шагам:

1. **Фаза Аналитики (BA)**
   - Прочти файл: `.agent/roles/ba.md`
   - Выполни действия, соответствующие этой роли (создай/обнови план).
   - **Если планируются изменения в архитектуре** - переходи к роли **Architect** или сразу к **Developer**, есть таких планов нет

2. **Фаза Архитектуры (Architect)** *если меняется архитектура*
   - Прочти файл: `.agent/roles/architect.md`
   - Измени абстракции, `ARCHITECTURE.md`, `c4_models.md`.
   - **ПЕРЕДАЙ** задачу разработчику (**Developer**)

3. **Фаза Разработки (Developer)**
   - Прочти файл: `.agent/roles/developer.md`
   - Напиши код, следуя наброскам архитектора и плану.
   - **ОСТАНОВИСЬ** и спроси: "Код написан. Требуются ли тесты или проверка? (Переход к **QA / Tester**)?". Если пользователь подтвердил тесты - переходи к **QA / Tester**. Если тесты не требуются - переходи к **DevOps**.  

4. **Фаза Тестирования (QA)** *(если пользователь запросил тесты)*
   - Прочти файл: `.agent/roles/tester.md`
   - Тщательно проверь написанный код, добавь юнит-тесты (если есть фреймворк), поищи Edge Cases.
   - **ПЕРЕХОДИ** к **DevOps**"

5. **Фаза DevOps** *(если требуется)*
   - Прочти файл: `.agent/roles/devops.md`
   - Внеси изменения в `DEPLOYMENT.md`, `Dockerfile`, `.env.template` и т.д.
   - Заверши пайплайн.

> **Примечание:** Пользователь может прервать конвейер в любой момент или пропустить фазу. Каждая роль — это отдельный контекст. Если вам предлагают "одеть шляпу Тестировщика", вы перестаете мыслить как "DevOps" и фокусируетесь только на поломках кода.
