# AI Agent Instructions

> **IMPORTANT:** These instructions are **mandatory** for all AI agents working on this project unless explicitly overridden in a specific request.

---

## 🔐 Security & Privacy

### Never Commit Secrets
**CRITICAL:** Never commit or expose in any documentation:
- API keys (Jira, TickTick, Telegram, OpenAI, etc.)
- Database URLs with passwords
- Access tokens
- Email addresses
- Account IDs
- Production credentials

**Use placeholders instead:**
```bash
JIRA_API_TOKEN='<YOUR_JIRA_API_TOKEN>'
JIRA_EMAIL='<YOUR_EMAIL>@gmail.com'
JIRA_ACCOUNT_ID='<YOUR_JIRA_ACCOUNT_ID>'
```

### Personal Information
- Replace real emails with `<YOUR_EMAIL>` or generic examples
- Replace domain names with `<YOUR_DOMAIN>.atlassian.net`
- Replace account IDs with `<YOUR_ACCOUNT_ID>`
- Never include real names, phone numbers, or addresses

## 🏗️ Architecture Principles

### Database as Single Source of Truth
- **PostgreSQL database** is the authoritative data source
- External systems (TickTick, Jira) are **synchronized reflections**
- All operations should update DB first, then sync externally

### Two-Way Synchronization
- When postponing tasks: update DB + TickTick + Jira
- When updating due dates: sync to all three systems
- Never leave data inconsistencies between systems

### Stateless Container Architecture
- Service runs in on-demand containers that suspend after inactivity
- **Never use:** `while(true)`, `setInterval`, infinite loops
- **For recurring tasks:** Use external cron jobs that call API endpoints

## 📝 Code Style & Standards

### TypeScript
- Use strict mode
- Prefer explicit types over `any`
- Use interfaces for DTOs and domain models
- Follow NestJS conventions (services, controllers, modules)

### Logging
- Use NestJS Logger: `this.logger.log()`, `this.logger.error()`
- Log important events: incoming webhooks, sync operations, errors
- Include context in logs: task IDs, Jira keys, operation types

### Error Handling
- Wrap external API calls in try-catch
- Log errors with full context
- Return user-friendly error messages
- Never expose internal errors to users

### Testing
- Write end-to-end tests for critical workflows
- Test locally before deploying
- Verify webhook endpoints manually
- Check production logs after deployment

## 🔄 Git & Version Control

### **CRITICAL: No Remote Git Operations**
- ❌ **NEVER** interact with remote Git repositories
- ❌ **NO** cloning, pulling, pushing, fetching, or adding remotes
- ❌ **NO** git commands that touch GitHub, GitLab, Bitbucket, etc.

**Why:** This project uses Abacus AI's internal versioning system. Remote git operations will corrupt the setup and break checkpoint/restore functionality.

**If user requests Git operations:**
1. Politely decline
2. Explain the internal versioning system
3. Suggest using the coding vertical in a new DeepAgent conversation if needed

### Commits & Checkpoints
- Save checkpoints after completing major features
- Use descriptive checkpoint messages (4-5 words)
- Test before saving checkpoint

## 🌐 External Integrations

### Jira
- Use Account ID for assignee checks (not email!)
- Handle ADF (Atlassian Document Format) for descriptions
- Always fetch full issue details from API, don't trust webhook payload
- Map priorities: Highest→5, High→3, Medium→1, Low/Lowest→0

### TickTick
- Auto-create "Jira" project if doesn't exist
- Set due dates to 23:59:00 in Kyiv timezone (UTC+2)
- Use format: `YYYY-MM-DDTHH:MM:SS+0200`
- Always sync tags: `['jira']`

### Telegram
- Remove inline keyboards after button click (prevent multiple clicks)
- Use Markdown formatting for messages
- Keep bot responses concise and user-friendly
- Show Jira keys without brackets in titles (e.g., "HOME-10 — Task name")

### LLM (RouteLLM)
- Use `gpt-4o-mini` for cost efficiency
- Keep system prompts clear and concise
- Handle LLM errors gracefully (fallback to defaults)
- Log LLM usage for debugging

## 🚨 Breaking Changes Protocol

### Database Schema Changes
1. Create Prisma migration
2. Test migration locally
3. Backup production database (if possible)
4. Deploy migration during low-traffic period
5. Verify data integrity after migration

### API Changes
1. Maintain backward compatibility when possible
2. Version breaking changes (e.g., `/v2/endpoint`)
3. Update Swagger documentation
4. Test all affected endpoints
5. Update clients (webhooks, Telegram commands)

## 🎯 Project-Specific Rules

### Morning Planning (10:00 Kyiv)
- Show estimated time for each task
- Show total time at bottom
- Display postponed tasks separately with reasons
- Include LLM motivational text
- Remove capacity selection buttons after click

### Evening Checkup (21:00 Kyiv)
- Combine all incomplete tasks in **one message**
- Use single button: "Перенести все на завтра"
- Separate handling for tasks with `postponed_count > 3`
- Remove buttons after click

### Task Postponement
- Update `postponed_count` in DB
- Update `due_date` in DB
- Sync to TickTick (update due date)
- Sync to Jira (update duedate field)
- Show confirmation with updated count

### New Task Notifications
- Send immediately (24/7, no time restrictions)
- Show Jira key, title, and LLM time estimate
- Only for **new** tasks (not updates)
